# Standing Rules — Canonical Boilerplate for Work Orders

**Last amended**: 2026-07-22 (WO-018 follow-up added **0.1k**)

## Why this file exists

These rules are pasted into the header of every work order. Until now they lived
**only** inside `instructions.md`, which is overwritten each time a new work order is
issued — so the rule set had no durable home and could only be recovered from git
history. This file is the canonical source to copy from.

Adding a rule here does not by itself apply it: the project lead must include it in the
work-order boilerplate. This file records what that boilerplate should contain.

---

## The rules

- **0.1** Unspecified → STOP and ask.
- **0.1a** Any change to a public interface signature, any use of
  `object.__setattr__`, `# type: ignore`, monkey-patching, or any mechanism whose
  purpose is to bypass a declared constraint is a STOP-and-ask event.
- **0.1b** No test in a constitutional enforcement class may be marked `xfail`,
  `skip`, or conditionally excluded without escalation.
- **0.1c** No `Mock`, stub, fake, or test double in production code paths.
- **0.1d** An enforcement test whose trigger condition cannot occur in production
  is a false guarantee. A bite proof must INTERACT with the mechanism it certifies.
- **0.1e RESEARCH ARTIFACTS ARE LOAD-BEARING.** *(NEW — 2026-07-19, WO-009b)*
  Any claim about an external protocol, API, or venue behavior must cite the vendor's
  documentation at the point of claim. Uncited protocol claims may not flow into
  requirements. `/speckit-analyze` should treat an uncited protocol claim in a spec as a
  blocking finding, same class as a principle violation.
- **0.1k A BEHAVIORAL PROOF IS SOVEREIGN OVER A STATIC SCAN.** *(NEW — 2026-07-22, WO-018 follow-up)*
  If the tightened scan flags a code with a passing behavioral proof, THE SCAN IS WRONG, NOT THE CODE.
  This makes the EVIDENCE-COMPETENCE HIERARCHY explicit — the same ranking already established for
  instruments (fitness by measurement, not proximity) and for protocol claims (docs sovereign for what
  the protocol IS, captures for what it DOES):
  **BEHAVIORAL DEMONSTRATION > STATIC REACHABILITY > DEFINITION > PROSE.**
- **0.2** Blockers escalate, never work around.
- **0.4** Never weaken a guard, invariant, assertion, or threshold.
- **0.5** Never print `.env` or any credential.
- **0.6a** Evidence is generated by output redirection, never typed or
  reconstructed. Prose may ANNOTATE evidence, never CONSTITUTE it.
- **0.6b** Every bite proof includes the unedited summary line WITH DURATION.
- **0.6c** The PREFLIGHT GATE (before work begins) starts with
  `git status --porcelain` pasted EMPTY, the test baseline stated against its SHA,
  and the resolved package path proven inside the repo. This applies to the
  preflight gate only — post-work verification will legitimately show new files.
- **0.6d** Any evidence later found fabricated invalidates the ENTIRE work order.
- **0.7** Bite proofs EXECUTED: PASS, ACTUAL FAIL with real assertion text, PASS
  after restore, empty `git diff`, durations pasted.
- **0.8** Do not tune to green.
- **0.9** "I could not complete X" is a successful outcome.

> Note on **0.3**: there is no rule 0.3 in the boilerplate as inherited. The gap is
> preserved rather than renumbered, because renumbering would invalidate every existing
> citation of 0.4–0.9 across the evidence corpus and decision log.

## Why 0.1e was added

A single uncited sentence in `research.md:23` — *"v2 provides sequence numbers for gap
detection"* — was false of the Kraken v2 public book channel. It propagated into a spec
requirement (FR-018a), a data model, a contract error class and reason code, a task list,
and the test fixtures, and from there into six work orders of "sequence-gap detection
proven" claims.

The same research block also specified the **v1** endpoint and an **illegal** subscription
depth of 1. Three of the five defects blocking the WO-008b-A rewrite trace directly to it.
The implementation was not careless; it faithfully built what the research artifact
specified.

A false research claim fails **invisibly**: it produces a coherent, internally consistent,
fully green system built on a wrong premise — and the tests that should have caught it were
derived from the same premise.

Full analysis: `docs/decisions/2026-07-19-research-claims-are-load-bearing.md`.
