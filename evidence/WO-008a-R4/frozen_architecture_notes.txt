QUESTIONS FOR PROJECT LEAD:
---------------------------
1. Which component now OWNS cost computation — the loop, or execution?
   ANSWER: The loop (live.py, backtest/runner.py) now owns cost computation via CostModel.calculate_costs_from_market_state().
   Execution (paper.py) receives the pre-computed spread_cost as a parameter.

2. Does this interface remain venue-neutral for Sprint 3 Kraken/Coinbase swap (Principle VII)?
   ANSWER: POTENTIAL VIOLATION. A new venue adapter would be forced to accept a pre-computed cost it cannot influence.
   The venue adapter might need different cost calculation (e.g., different fee structures, spread calculations).

3. What alternatives existed?
   ALTERNATIVE 1: Execution receives MarketState and computes cost itself
   - pro: Venue controls its own cost calculation
   - pro: Interface remains venue-neutral (only MarketState passed)
   - con: Execution layer needs MarketState dependency

   ALTERNATIVE 2: Pass a CostComponent object instead of scalar spread_cost
   - pro: Structured cost data
   - pro: Venue can inspect/modify the full cost breakdown
   - con: More complex interface

3.2 object.__setattr__ bypassing frozen dataclass in live.py
------------------------------------------------------------

CODE (R3):
```python
# src/trading/loop/live.py lines 337-354
def _update_position(self, fill: dict) -> None:
    """Update position state after fill."""
    size = Decimal(str(fill["size"]))
    side = fill["side"]
    total_cost = Decimal(str(fill["total_cost"]))

    if side == "BUY":
        new_quantity = self._position_state.current_quantity + size
    else:
        new_quantity = self._position_state.current_quantity - size

    # Update realized P&L (simplified)
    new_pnl = self._position_state.realized_pnl - total_cost

    # Use object.__setattr__ to bypass frozen dataclass restriction
    object.__setattr__(self._position_state, 'current_quantity', new_quantity)
    object.__setattr__(self._position_state, 'realized_pnl', new_pnl)
```

FROZEN CLASS DEFINITION:
------------------------
```python
# src/trading/risk/position_state.py
@dataclass(frozen=True)
class PositionState:
    """Current trading position state."""
    symbol: str
    current_quantity: Decimal
    average_entry_price: Decimal
    unrealized_pnl: Decimal
    realized_pnl: Decimal
    daily_pnl: Decimal
```

QUESTIONS FOR PROJECT LEAD:
---------------------------
1. What class is being mutated, and why was it declared frozen?
   ANSWER: PositionState is a frozen dataclass declared with @dataclass(frozen=True).
   It was likely declared frozen for immutability guarantees (e.g., audit trail determinism).

2. Does mutating it affect determinism?
   ANSWER: POTENTIAL ISSUE. test_determinism_verified is one of the 11 tests just un-xfailed.
   Mutating frozen state could affect the ability to reconstruct state from logs (Principle VIII).

3. Does it affect the audit/decision log (Principle VIII)?
   ANSWER: The mutation happens AFTER the decision is logged, so the decision log itself is not affected.
   However, the position state between decisions might not be reconstructible if frozen state is mutated.

4. What alternatives existed?
   ALTERNATIVE 1: dataclasses.replace() returning a new instance
   - pro: Respects frozen contract
   - pro: Immutable by design
   - con: Requires reassigning self._position_state everywhere

   ALTERNATIVE 2: Explicitly mutable state container
   - pro: No frozen bypass needed
   - con: Loses frozen dataclass benefits

   ALTERNATIVE 3: Redesign update path to use immutable updates
   - pro: Cleanest architecture
   - con: More invasive change
