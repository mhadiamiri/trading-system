═══════════════════════════════════════════════════════════════════════════
A.1 THRESHOLDS — numeric, with derivations
═══════════════════════════════════════════════════════════════════════════
PING_SAMPLE_INTERVAL = 1 s
  A protocol control frame, not an application message. ~60 RTT samples/min resolves a
  distribution; sparse against the 38 ms frame cadence so it is not itself load. 1 ping/s
  against ~26 msg/s inbound is negligible; Kraken publishes no ping rate limit (documented
  silence) — noted so no one later suspects the probe perturbed the measurement.

LAG_SAMPLE_INTERVAL = 100 ms
  Samples the loop ~10x/s (~600 samples/min) — fine enough to catch sub-second starvation
  bursts, coarse enough not to be its own load.

"LATE" pong: rtt > 250 ms
  Justified against EXPECTED RTT, not as a round number. We hold NO measured RTT to Kraken's
  WS endpoint (NO VENUE CONNECTION this WO), so this is stated as conservative. A client to
  a major-venue WS endpoint is plausibly 20–100 ms round-trip (typical cross-region internet
  RTT); 250 ms is therefore ~2.5–10x nominal — above ordinary network variance, and reachable
  when the event loop needs >~200 ms to service the received pong. The figure only FLAGS an
  abnormal RTT; the LAG sampler, not this number, decides WHY it is late (network/venue vs
  starvation). Conservative by intent: a higher threshold risks missing genuine lateness, a
  lower one risks flagging ordinary jitter — 250 ms sits above plausible jitter.

"ABSENT" pong: no matching pong within 5 s of the ping
  5 s == the application-ping interval (WO-014b-2 §1.2) and is >> any plausible RTT plus
  reasonable starvation. A pong unseen at 5 s is effectively gone. (Absent is the limiting
  case of late — same causal reading, see A.3.)

"ELEVATED" lag: measured lag > 100 ms on > 5 % of samples
  100 ms of lag on a 100 ms sampling interval means a FULL sampling interval was consumed
  before the sentinel task could run again — a meaningful block against the ~38 ms/frame
  budget at 26 msg/s (a >100 ms stall drops ~3 frames' worth of scheduling). The > 5 %
  fraction demands SUSTAINED elevation across the window, not a lone GC pause or one-off
  stall.

"GAPPY" instruments (VOID): missed-sample fraction > 10 % of expected, on EITHER the lag
  sampler or the pong observer
  expected = window_seconds / sample_interval. Losing > 10 % of expected samples means the
  instrument was itself starved enough that its surviving record is a biased sample (see
  A.3 / Branch 5). > 10 % is the point past which threshold comparisons cannot be trusted.

═══════════════════════════════════════════════════════════════════════════
A.2 THE SHARED CLOCK — hard requirement
═══════════════════════════════════════════════════════════════════════════
All three instrument records — PONG LATENCY, LAG SAMPLES, and MESSAGE COUNTS (and the
receive-to-process latency) — MUST timestamp from the SAME CLOCK SOURCE:

    time.monotonic()   — NOT wall clock (datetime / time.time()).

Declared here in the WO record, and to be repeated in EACH instrument's docstring in Phase
B. Reason: the entire discrimination is a CORRELATION argument across records (do lag gaps
coincide with message-rate peaks? do late pongs coincide with elevated lag?). Correlation
across mixed clock bases is a quiet methodological hole that survives until someone tries to
use it at hour nine — cheap to declare now, impossible to repair after. monotonic() is also
immune to wall-clock steps (NTP adjustments) that would corrupt sub-second correlation.

A.3 CROSS-PRODUCT ENUMERATION — every cell mapped
═══════════════════════════════════════════════════════════════════════════
STATE SPACES:
  pong state ∈ { ON-TIME (rtt ≤ 250 ms), LATE (rtt > 250 ms), ABSENT (no pong ≤ 5 s) }
  lag state  ∈ { NORMAL (not elevated), ELEVATED (> 100 ms on > 5 % of samples) }
  Overriding: GAPPY (either instrument > 10 % missed) — a THIRD axis that voids the above.

FULL CROSS-PRODUCT (3 pong × 2 lag = 6 cells) — NO cell left unmapped:

  ┌───────────┬──────────────── lag NORMAL ─────────────┬──────────── lag ELEVATED ───────────┐
  │ ON-TIME   │ Branch 4: both hypotheses FALSIFIED      │ Branch 2: STARVATION                │
  │           │  (+ no fault). STOP before a third.      │  (remedy = pipeline architecture)   │
  ├───────────┼─────────────────────────────────────────┼─────────────────────────────────────┤
  │ LATE      │ Branch 1: protocol/venue; 1011 SURVIVES  │ Branch 3: STARVATION-PRIMARY;       │
  │           │                                          │  protocol retained OPEN (not falsi.)│
  ├───────────┼─────────────────────────────────────────┼─────────────────────────────────────┤
  │ ABSENT    │ Branch 1: protocol/venue; 1011 SURVIVES  │ Branch 3: STARVATION-PRIMARY;       │
  │           │  (absent = limiting case of late)        │  protocol retained OPEN             │
  └───────────┴─────────────────────────────────────────┴─────────────────────────────────────┘

CELL-BY-CELL, with justification:
  (ON-TIME , NORMAL)  → BRANCH 4. Pongs answered on time AND the loop is not starved AND no
      fault occurred → both the protocol-timeout and the starvation hypotheses are falsified.
      STOP before theorizing a third cause. QUALIFIER on "no fault": with ping_timeout=None
      there is no 1011; a venue-initiated close would arrive via §1.3's venue-close path. If a
      close DID occur in this cell, it is NOT Branch 4 — it is an unexplained fault to report,
      not a clean falsification. Branch 4 requires on-time + normal + no fault.
  (ON-TIME , ELEVATED) → BRANCH 2. Loop starved but Kraken still answers our probes on time →
      the fault is on OUR side (scheduling), remedy is pipeline architecture, not protocol
      compliance. The ≥60/min throughput verdict must CARRY this finding: starvation at one
      hour predicts worse at twenty-four.
  (LATE , NORMAL)     → BRANCH 1. Pongs slow but the loop is healthy → the delay is not ours;
      protocol/venue issue, the original 1011 hypothesis survives.
  (ABSENT , NORMAL)   → BRANCH 1. Same reading as (LATE, NORMAL): a healthy loop did not
      cause the missing pong, so it is a protocol/venue issue. Absent is the limiting case of
      late — identical causal direction — so it shares the branch. Mapped explicitly, not
      left as a gap.
  (LATE , ELEVATED)   → BRANCH 3. STARVATION-PRIMARY. The causal arrow runs ONE way: elevated
      lag CAN produce late pongs (a starved loop services the pong slowly); late pongs CANNOT
      produce elevated lag. So the elevated lag is the primary cause and the protocol
      hypothesis is retained as OPEN, not falsified.
  (ABSENT , ELEVATED) → BRANCH 3. Same one-way causal logic: a loop starved enough to elevate
      lag can starve the pong entirely (absent). Starvation-primary, protocol retained OPEN.
      Mapped explicitly — this cell is not in the five branches verbatim, and leaving it
      unmapped is exactly the four-branches-with-a-gap that produced S13.

EXCLUDED CELLS: NONE. All 6 pong×lag cells map to a branch. The only states not appearing as
their own cell are folded deliberately: ABSENT reuses LATE's branches (limiting case, same
causal direction, justified above).

BRANCH 5 — GAPPY — OVERRIDES ALL OF THE ABOVE when it applies:
  If either instrument is GAPPY (> 10 % missed samples), the run is VOID regardless of which
  cell its (biased) surviving numbers would land in. Ruled interpretation:
  - VOID attaches to the QUANTITATIVE discrimination. A gappy record is a biased sample —
    recorded preferentially during HEALTHY periods (the sampler runs when the loop is free) —
    so NO threshold comparison survives that bias. No branch verdict.
  - THE GAPPINESS ITSELF IS A REPORTED FINDING. The missed-sample count and the gap
    timestamps are output, not absence of output.
  - PRE-RULED CORRELATION: if lag-sampler gaps coincide with MESSAGE-RATE PEAKS on the shared
    clock (A.2), that coincidence is ADMISSIBLE EVIDENCE FOR the starvation hypothesis and
    supports a directed follow-up — but it is NEVER a branch verdict.
  - GOVERNING DISTINCTION: gappy instruments can NOMINATE a hypothesis; only CLEAN
    instruments can CONVICT one. A VOID run reports: "no branch verdict, PLUS this correlated
    evidence, PLUS this nomination" — not "no result."

This is why Branch 5 needs the shared clock (A.2): a nomination is only makeable if the gap
timestamps are correlatable against the message-rate and pong records on one clock.
