################ ⚠ HONEST LIMIT — STATED PLAINLY AS REQUIRED ################

THE SNAPSHOT PATH IS INDEPENDENTLY VERIFIED. Kraken publishes checksum
3310070434 for a documented 10-level book, and our implementation reproduces
it. The expected value originates with Kraken, not with us.

THE INCREMENTAL POST-UPDATE ORDERING IS **NOT** VERIFIED, AND CANNOT BE BY
ANY FIXTURE. Kraken publishes no incremental example with a checksum. Any
fixture we build encodes OUR OWN ordering assumption on both sides of the
comparison — we compute the expected value with the same code under test, so
a shared misunderstanding would agree with itself and pass.

NO FIXTURE WAS MANUFACTURED TO APPEAR TO VERIFY IT. What the new test
test_checksum_is_validated_after_applying_not_before DOES establish is
narrower and is labelled as such in its own docstring: that the pre-update
and post-update ladders yield DIFFERENT checksums, and that the declared
value matches the POST-update one. That demonstrates which ordering is in
force. It does not prove that ordering is what Kraken means.

THIS FIX IS VERIFIABLE ONLY AT FIRST LIVE CONTACT — WO-008b-A2.
If it is wrong, the symptom will be immediate and loud: every incremental
mismatches, consecutive_failures climbs, and the adapter enters a reconnect
loop emitting nothing. That is a legible failure, not a silent one.
