WO-014b-2 ADDENDUM — GAP-RECORD ATTACHMENT POINTS (for WO-014c schema design)
Generated: 2026-07-20.  Baseline: bdb5052 (WO-014b-2 §1 complete).

PURPOSE (project-lead ruling): WO-014c will record data gaps against a DECLARED schema
and cause taxonomy — keepalive reconnect, checksum resync, breaker-adjacent retry ladder,
venue disconnect. Two of those causes are created by the §1 slice just shipped. This
document names, for each gap-producing path in the shipped implementation, the EXACT code
location where a gap record would be emitted and WHAT IS IN SCOPE there — so WO-014c can
specify the schema against real emission sites rather than retrofitting to whatever the
capture happened to write (the "fixtures-shaped-to-the-implementation, one layer up" trap).

NO GAP RECORDING IS IMPLEMENTED HERE. This is a reporting/spec artifact only. All line
numbers are src/trading/data/adapters/kraken_v2_book.py @ bdb5052.

Convention: a "gap" is a [t_open, t_close] interval during which NO validated MarketState
is emitted. Each path below is described as (OPEN site, CLOSE site, in-scope fields).

CAUSE 3 — BREAKER-ADJACENT RETRY LADDER (backoff)        [cause: reconnect_retry_ladder]

EMISSION site: _perform_reconnect backoff loop — line 1157 `self._reconnect_ladder.append(
            {"attempt","at","delay_s","error"})`, one entry per FAILED reopen. This is the
            DURATION detail of any reconnect gap (CAUSE 1, 2, or 4): the ladder's span is

IN SCOPE (per attempt, line 1157-1162):

NOTE: a successful reconnect's ladder is currently RETAINED on the instance but not emitted

CAUSE 4 — VENUE DISCONNECT                               [cause: venue_disconnect]
════════════════════════════════════════════════════════════════════════
Three sub-paths, with DIFFERENT current coverage — flagged honestly:

4a. TERMINAL (breaker trip = venue presumed gone). BEST-DEFINED emission site already:
    OPEN/record: _trip_circuit_breaker — line 1203 `self.capture_terminated = {...}`, raised
    at line 1146. capture_terminated ALREADY carries the full terminal-gap record:
    reason_code, trip_time, trigger, retry_ladder, last_validated_book, frames_captured,
    evidentiary_bounds. **WO-014c's gap schema should be shaped to MATCH capture_terminated**
    — it is the one place a gap record is fully assembled today, and the run STOPS here.

4b. APPLICATION-PING-SEND FAILURE (socket dropped, detected on send). Emission site:
    get_live_market_data line 1338-1343 (`except Exception ... _perform_reconnect(reason=
    "application ping send failed")`). Same in-scope fields as CAUSE 1 (a reconnect gap).

4c. EXPLICIT VENUE CLOSE ON recv (ConnectionClosed frame). **COVERAGE GAP — FLAG.** The
    recv at line 1362 catches ONLY asyncio.TimeoutError (1363). A websockets ConnectionClosed
    raised by recv PROPAGATES out of get_live_market_data and ENDS THE CAPTURE — there is no
    reconnect and no gap record. Silent drops are covered by heartbeat-absence (CAUSE 1), and
    reopen failures by backoff, but an EXPLICIT close frame is not routed through the reconnect
    path today. This is the residual shape of the §0 hard-stop for the explicit-close case.
    RECOMMENDATION (NOT implemented — production change, needs approval): wrap recv to route
    ConnectionClosed through _perform_reconnect (reason "venue closed connection"), giving 4c
    the same gap attachment as 4a/4b. Until then, WO-014c should treat 4c as an uncovered
    emission site.
