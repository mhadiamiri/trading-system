A.2 — STANDING REPORT FORM (every future delta): SIGNAL | NOISE FLOOR | RATIO
  SIGNAL     = |new_mean_cycle - baseline_mean_cycle|
  NOISE      = the declared floor (operational: 1.0 ms cross-session on the current instrument)
  RATIO      = SIGNAL / NOISE
  RULE: RATIO < 1  =>  delta is INSIDE the noise floor: record it WITH SIGN EXPLICITLY UNESTABLISHED,
        and KEEP the ledger entry ("changed X, could not detect a cost" BOUNDS the effect — which is
        what the feasibility clause needed). RATIO >= ~2-3 => sign established; report signal+floor+ratio.

  Worked example — WO-017's headline delta re-read under this form:
     SIGNAL = 0.963 ms ; NOISE = 1.0 ms ; RATIO = 0.96  =>  INSIDE THE FLOOR, SIGN UNESTABLISHED.

MECHANICS: replay the pinned WO-009 window through the OLD and NEW code paths ALTERNATELY, in the
SAME process where possible, ORDER RANDOMIZED per existing doctrine (paired trials, A/B/B/A... with a

DOES INTERLEAVING BUY THE RESOLUTION WE NEED? — from the data above:
  - Cross-session offset (the term interleaving cancels) = ~1.0 ms.
  - Within-session spread (the floor interleaving is LIMITED BY) = ~0.3 ms (1 sigma), ~0.6-0.8 ms p2p.
  Because within-session spread (~0.3 ms) is materially SMALLER than the cross-session offset (~1.0 ms),
  interleaving WOULD improve resolution: the floor drops from ~1.0 ms (cross-session) to ~0.3-0.6 ms
  (within-session). WO-017's 0.963 ms delta sits ABOVE a ~0.3-0.6 ms within-session floor, so an
  interleaved A/B could plausibly ESTABLISH its sign where the cross-session comparison cannot.
  Same-process interleaving may tighten this further (shared warmup/JIT/page-cache state), so ~0.3 ms
  is an upper bound on the achievable floor.

COST: retain BOTH code paths simultaneously for the arms (a git-worktree/checkout A/B harness, or a
runtime switch preserving the old path), plus paired-trial orchestration and a seeded shuffle. Modest
one-time tooling; each A/B run costs ~2x a single establishment (both arms). No venue connection.

CONCLUSION (either outcome is fine; unquantified resolution is not): the numbers SUPPORT adoption for
deltas >= ~0.5-0.6 ms — interleaving removes the ~1.0 ms cross-session term and lowers the floor to the
within-session spread. For SUB-0.5 ms deltas on this host, even interleaving is at/below floor: declare
them below the instrument's resolution rather than reporting a sign. Recommended adoption: use
interleaved A/B whenever a pipeline change's expected per-frame delta is in the 0.5-1.0 ms band; skip it
(declare below-floor) for sub-0.5 ms expected effects. Pre-approved — NOT implemented here.
