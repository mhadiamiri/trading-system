§1.2 DIAGNOSIS — Is spread double-counted in P&L?
--------------------------------------------------

ANSWER: YES

EVIDENCE: evidence/WO-008a-R6/double_count_diagnosis.txt

ARITHMETIC PROOF:
- BUY at ask 65980.0 for 0.1 BTC: cash outflow = 6598.0
- Executed price already includes spread cost (0.25)
- Gross P&L calculation: -6598.0 (includes spread)
- Old total_cost: 6.598 (fees) + 0.25 (spread) + 6.598 (slippage) = 13.446
- Net P&L: -6598.0 - 13.446 = -6611.446

RECONCILIATION:
- Correct economic cost: 6597.75 (mid) + 6.598 (fees) + 6.598 (slippage) = 6610.946
- Current overstatement: -6611.446
- Difference: 0.25 (exactly the spread counted twice)

§1.3 RESOLUTION — Spread as Attribution
---------------------------------------

CHOICE: (A) PREFERRED

WHY:
- Executing at ask/bid is realistic (real venues match orders at bid/ask)
- Executed price naturally includes the spread cost
- Spread reported transparently as attribution, not hidden
- Total cost = fees + slippage (actual additive costs only)

RECONCILIATION ARITHMETIC:
```
BUY 0.1 BTC at ask (65980.0), bid 65975.0, spread 5.0

Executed price includes spread:
- BUY pays ask: 0.1 × 65980.0 = 6598.0
- If at mid: 0.1 × 65977.5 = 6597.75
- Spread in executed price: 6598.0 - 6597.75 = 0.25 ✓

Additive costs:
- Fees: 6.598
- Slippage: 6.598
- Total additive: 13.196

Total economic cost:
- Executed price (6598.0) + Additive costs (13.196) - Spread already in executed (0.25)
- = 6598.0 + 13.196 = 6611.196 ✓
- OR: Mid (6597.75) + Fees (6.598) + Slippage (6.598) + Spread (0.25) = 6611.196 ✓

Nothing counted twice: spread appears once as executed price impact, reported
transparently as attribution, NOT added to total_cost.
```

§2.1-2.2 STALENESS GUARD SPECIFICATION
--------------------------------------

BEHAVIOR:
- No MarketState set → place_order() RAISES with EXEC_NO_MARKET_STATE
- MarketState stale beyond threshold → place_order() RAISES with EXEC_STALE_MARKET_STATE

REASON CODES (consistent with existing convention):
- EXEC_NO_MARKET_STATE (no state registered)
- EXEC_STALE_MARKET_STATE (state too old)
- Pattern: EXEC_* for execution layer (matches EXEC_ORDER_FILLED, EXEC_BLOCKED_KILL_SWITCH)

THRESHOLD VALUE: 18 seconds

WHERE CONFIGURED:
- DEFAULT_STALENESS_THRESHOLD_SECONDS = 18 (paper.py line 48)
- Constructor parameter: staleness_threshold_seconds (configurable)
- Stored as: self._staleness_threshold = timedelta(seconds=staleness_threshold_seconds)

JUSTIFICATION:
- Historical rate: ~10 MarketStates/min → 1 state every 6 seconds
- Threshold: 18 seconds = 3x historical interval
- Derivation: 3 × (60 / 10) = 18 (not a round number picked arbitrarily)
- Allows normal variance (brief gaps common in WebSocket feeds)
- Detects genuine stalls within seconds
- For WO-008b: acceptance threshold ≥60/min, historical ~10/min
- 18s threshold = 3.3 states/min minimum, well below acceptance but detects stalls
