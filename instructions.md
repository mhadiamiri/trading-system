## WHY THIS EXISTS

The Kraken v2 public book channel carries a CRC32 checksum per update and NO
sequence number. FR-018a mandates sequence tracking — a mechanism that does not
exist on this channel. The fixtures carried a synthetic `sequence` field, so
"sequence-gap detection proven" was a guarantee attached to a fictional protocol.

Root cause, from the diagnosis: **"fixtures supply QuoteUpdate objects rather
than raw frames — the fixtures were shaped to fit the implementation rather than
the protocol."** While that holds, the parse path is not under test at all and
fixtures will keep drifting toward whatever the implementation expects. This work
order attacks the generator, not its products.

## 1. AMEND THE SPEC

Edit `specs/002-quote-level-data/spec.md`.

### 1.1 FR-018a — replace the sequence-number mandate
The amended FR must state ALL of:
- **Integrity mechanism:** CRC32 checksum, not sequence numbers. The v2 public
  book channel provides no sequence numbers.
- **Ordering — this IS the guarantee, not an implementation detail:** the CRC32
  checksum is defined over the **post-update book state**; validation therefore
  occurs **after** applying each update, and a mismatch invalidates the applied
  state.
- **Validation frequency: EVERY update.** Kraken permits periodic validation;
  permitted and honest are different standards. Put the every-update mandate in
  the FR text so it cannot later drift to "periodic" as an optimization.
- **Full recovery semantics:** checksum failure → discard local book →
  resubscribe / request fresh snapshot → **emit NO MarketState from the moment of
  failure until the fresh snapshot is applied and its checksum validates.** A book
  in an unverified state must not price anything (Principle V).
- **Subscription depth:** name the legal Kraken values (10, 25, 100, 500, 1000)
  and PIN the project's depth as a stated fact. `BOOK_DEPTH = 1` was only possible
  because no spec text made depth explicit. State the value chosen and why.

### 1.2 Remove `sequence` from the specified `QuoteUpdate` schema
No vestigial fields modeling a fictional protocol.

## 2. REBUILD FIXTURES AS RAW v2 FRAMES

### 2.1 Required shape
New fixtures are **raw v2 dict envelopes matching Kraken's documented schema** —
what the WebSocket actually delivers — not pre-parsed `QuoteUpdate` objects.
Full field set required:
- `channel` ("book"), `type` ("snapshot" | "update"), `symbol`
- `bids` / `asks` as **arrays of `{price, qty}`** at a legal depth (not 4 scalars,
  not top-of-book only)
- server-sent `timestamp` (NOT client-generated `datetime.now(UTC)`)
- `checksum`

### 2.2 Coverage required
- **Snapshot vs update distinguishable** — current fixtures omit `type` entirely.
- **`qty: 0` deletion semantics EXERCISED** — Kraken's primary level-removal
  mechanism, currently never tested.
- **Book truncation to subscribed depth** after update (Kraken does not send
  `qty: 0` for levels falling out of scope).

`compute_checksum()` over a 1-level book. That proof is circular.

  truth**. Extract as many documented values as available.
- **Label every fixture:** `# GROUND TRUTH: Kraken docs <url>` versus
