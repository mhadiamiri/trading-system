Here are my answers to all five clarification questions. Integrate them into specs\001-walking-skeleton\spec.md in the Clarifications section, and update/add functional requirements as needed. Keep everything behavioral — config mechanism details belong in plan.md, not here.
Q1 — Pair: A (BTC/USD). Deepest liquidity gives the most realistic spread/slippage modeling.
Q2 — Max position: D (configurable, default 1 BTC). Also add: the risk-engine tests MUST include a case where the configured limit is small enough that a clamp actually fires, so clamp behavior is exercised in Sprint 1 rather than left untested.
Q3 — Max daily loss: D (configurable), but change the framing. Do NOT hardcode "$500" against an assumed $10,000 account — we never committed to that capital figure. Express the limit as a percentage of configured account equity, defaulting to 5%, with account equity itself a configured value. Rationale: when Sprint 3 begins with a few hundred real dollars, a fixed $500 limit would silently become a >100% daily-loss limit. A percentage scales correctly.
Q4 — Fee: D (configurable, default 0.1% taker per side). Taker, not maker: a seconds-to-minutes momentum/volume strategy crosses the spread, so assuming maker fees would understate costs and produce a dishonest backtest. Configurable because the fee schedule changes when the venue changes.
Q5 — Minimum backtest data: D (no minimum — the backtest runs on whatever stored data exists). Rationale: any minimum would block P2 behind a data-collection wait, which is a stall. P2's purpose is to prove the cost machinery is honest (fees + spread + slippage applied to every trade, P&L net of all three), NOT to produce a statistically meaningful edge estimate — that is a Sprint 2 walk-forward concern.
Add a guard requirement: every backtest result MUST report the data window it ran on — start timestamp, end timestamp, and number of events/trades — so no result can be quoted without its sample size attached.
